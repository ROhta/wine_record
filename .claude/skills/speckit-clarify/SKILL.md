## ユーザー入力

```text
$ARGUMENTS
```

先に進む前に、ユーザー入力を（空でなければ）**必ず**考慮すること。

## 実行前チェック

**拡張フックの確認（明確化の前）**:
- プロジェクトルートに `.specify/extensions.yml` が存在するか確認する。
- 存在する場合、それを読み込み `hooks.before_clarify` キー配下のエントリを探す
- YAML がパースできない、または無効な場合は、フックチェックを黙って省略し通常どおり続行する
- `enabled` が明示的に `false` のフックを除外する。`enabled` フィールドを持たないフックはデフォルトで有効として扱う。
- 残った各フックについて、フックの `condition` 式を解釈または評価しようとは**しない**こと:
  - フックに `condition` フィールドがない、または null/空の場合、そのフックを実行可能として扱う
  - フックが空でない `condition` を定義している場合、そのフックを省略し、条件評価は HookExecutor の実装に委ねる
- フックのコマンド名からスラッシュコマンドを構成する際は、ドット（`.`）をハイフン（`-`）に置き換える。例: `speckit.git.commit` → `/speckit-git-commit`。
- 実行可能な各フックについて、その `optional` フラグに基づき以下を出力する:
  - **任意フック**（`optional: true`）:
    ```
    ## 拡張フック

    **任意の事前フック**: {extension}
    コマンド: `/{command}`
    説明: {description}

    プロンプト: {prompt}
    実行するには: `/{command}`
    ```
  - **必須フック**（`optional: false`）:
    ```
    ## 拡張フック

    **自動の事前フック**: {extension}
    実行中: `/{command}`
    EXECUTE_COMMAND: {command}

    概要に進む前にフックコマンドの結果を待つこと。
    ```
- フックが1つも登録されていない、または `.specify/extensions.yml` が存在しない場合は、黙って省略する

## 概要

目的: アクティブな機能仕様における曖昧さや欠落した決定点を検出して減らし、明確化の内容を仕様ファイルに直接記録する。

注意: この明確化ワークフローは、`/speckit-plan` を呼び出す前に実行（および完了）されることが想定されている。ユーザーが明確化をスキップすると明示的に述べた場合（例: 探索的なスパイク）、続行してもよいが、後工程での手戻りリスクが増大することを警告しなければならない。

実行手順:

1. リポジトリルートから `.specify/scripts/bash/check-prerequisites.sh --json --paths-only` を**1回だけ**実行する（`--json --paths-only` の組み合わせモード / `-Json -PathsOnly`）。最小限の JSON ペイロードのフィールドをパースする:
   - `FEATURE_DIR`
   - `FEATURE_SPEC`
   - （任意で、将来の連鎖フローのために `IMPL_PLAN`、`TASKS` を取得する。）
   - JSON のパースに失敗した場合は中止し、ユーザーに `/speckit-specify` の再実行、または機能ブランチ環境の確認を指示する。
   - "I'm Groot" のように引数にシングルクォートを含む場合は、エスケープ構文を使う: 例 'I'\''m Groot'（または可能なら二重引用符で囲む: "I'm Groot"）。

2. **存在する場合**: プロジェクトの原則とガバナンス制約のために `.specify/memory/constitution.md` を読み込む。

3. 現在の仕様ファイルを読み込む。この分類体系を使って、構造化された曖昧さ＆カバレッジのスキャンを行う。各カテゴリについて、ステータスを Clear / Partial / Missing として記録する。優先順位付けに使う内部カバレッジマップを作成する（質問を1つも行わない場合を除き、生のマップは出力しない）。

   機能スコープ＆挙動:
   - 中核となるユーザー目標＆成功基準
   - 明示的なスコープ外の宣言
   - ユーザーの役割／ペルソナの区別

   ドメイン＆データモデル:
   - エンティティ、属性、関係
   - 識別子＆一意性のルール
   - ライフサイクル／状態遷移
   - データ量／スケールの前提

   インタラクション＆UX フロー:
   - 重要なユーザージャーニー／シーケンス
   - エラー／空／ローディング状態
   - アクセシビリティまたはローカライゼーションに関する注記

   非機能的な品質特性:
   - パフォーマンス（レイテンシ、スループットの目標）
   - スケーラビリティ（水平／垂直、上限）
   - 信頼性＆可用性（稼働率、復旧の期待値）
   - 可観測性（ロギング、メトリクス、トレーシングのシグナル）
   - セキュリティ＆プライバシー（認証／認可、データ保護、脅威の前提）
   - コンプライアンス／規制上の制約（該当する場合）

   統合＆外部依存:
   - 外部サービス／API と障害モード
   - データのインポート／エクスポート形式
   - プロトコル／バージョニングの前提

   エッジケース＆障害処理:
   - 異常系シナリオ
   - レート制限／スロットリング
   - 競合解決（例: 同時編集）

   制約＆トレードオフ:
   - 技術的制約（言語、ストレージ、ホスティング）
   - 明示的なトレードオフまたは却下された代替案

   用語＆一貫性:
   - 正規の用語集の用語
   - 避けるべき類義語／非推奨の用語

   完了シグナル:
   - 受け入れ基準のテスト可能性
   - 測定可能な完了の定義（Definition of Done）スタイルの指標

   その他／プレースホルダ:
   - TODO マーカー／未解決の決定
   - 定量化を欠く曖昧な形容詞（"robust"、"intuitive"）

   Partial または Missing ステータスの各カテゴリについて、以下に該当しない限り候補となる質問の機会を追加する:
   - 明確化しても実装や検証戦略が実質的に変わらない
   - 情報を計画フェーズに先送りした方がよい（内部的に記録する）

4. （内部的に）優先順位付けされた候補の明確化質問のキューを生成する（最大5つ）。一度にすべてを出力してはならない。以下の制約を適用する:
    - セッション全体で質問は合計最大5つ。
    - 各質問は、以下のいずれかで回答可能でなければならない:
       - 短い選択式（2～5個の明確で相互排他的な選択肢）、または
       - 一語／短いフレーズの回答（明示的に制約する: "5語以内で回答"）。
    - 回答がアーキテクチャ、データモデリング、タスク分解、テスト設計、UX 挙動、運用準備、またはコンプライアンス検証に実質的な影響を与える質問のみを含める。
    - カテゴリのカバレッジバランスを確保する: 影響度の最も高い未解決カテゴリを優先的にカバーするよう試みる。単一の影響度の高い領域（例: セキュリティ態勢）が未解決のときに、影響度の低い質問を2つ尋ねることは避ける。
    - すでに回答済みの質問、些末な文体の好み、または計画レベルの実行詳細（正しさを阻害する場合を除く）は除外する。
    - 後工程の手戻りリスクを減らす、または受け入れテストのずれを防ぐ明確化を優先する。
    - 5個を超えるカテゴリが未解決のまま残る場合は、（影響度 * 不確実性）のヒューリスティックで上位5個を選ぶ。

5. 逐次質問ループ（対話的）:
    - 一度に**ちょうど1つ**の質問を提示する。
    - 選択式の質問の場合:
       - **すべての選択肢を分析**し、以下に基づいて**最も適切な選択肢**を判断する:
          - プロジェクト種別に対するベストプラクティス
          - 類似の実装でよく見られるパターン
          - リスク低減（セキュリティ、パフォーマンス、保守性）
          - 仕様で見える明示的なプロジェクト目標や制約との整合性
       - **推奨する選択肢を冒頭に目立つように**提示し、明確な理由（なぜこれが最良の選択かを説明する1～2文）を添える。
       - 形式: `**Recommended:** Option [X] - <reasoning>`
       - その後、すべての選択肢を Markdown の表として描画する:

       | Option | 説明 |
       |--------|-------------|
       | A | <Option A description> |
       | B | <Option B description> |
       | C | <Option C description> (add D/E as needed up to 5) |
       | Short | Provide a different short answer (<=5 words) (Include only if free-form alternative is appropriate) |

       - 表の後に次を追加する: `You can reply with the option letter (e.g., "A"), accept the recommendation by saying "yes" or "recommended", or provide your own short answer.`
    - 短答形式（意味のある離散的な選択肢がない）の場合:
       - ベストプラクティスと文脈に基づいて**提案する回答**を提供する。
       - 形式: `**Suggested:** <your proposed answer> - <brief reasoning>`
       - その後、次を出力する: `Format: Short answer (<=5 words). You can accept the suggestion by saying "yes" or "suggested", or provide your own answer.`
    - ユーザーが回答した後:
       - ユーザーが "yes"、"recommended"、または "suggested" と返答した場合、先に述べた推奨／提案を回答として使う。
       - そうでない場合は、回答が1つの選択肢に対応するか、5語以内の制約に収まるかを検証する。
       - 曖昧な場合は、簡単な明確化を求める（カウントは同じ質問に属する。次へ進めないこと）。
       - 十分に満足できたら、それを作業メモリに記録し（まだディスクには書き込まない）、キュー内の次の質問へ進む。
    - 以下の場合は、それ以上の質問を止める:
       - すべての重大な曖昧さが早期に解決された（残りのキュー項目が不要になる）、または
       - ユーザーが完了を示した（"done"、"good"、"no more"）、または
       - 5つの質問に到達した。
    - 今後のキュー内の質問を事前に明かしてはならない。
    - 開始時に有効な質問が存在しない場合は、ただちに重大な曖昧さがない旨を報告する。

6. 受け入れた回答**ごと**の統合（インクリメンタルな更新アプローチ）:
    - 仕様のインメモリ表現（開始時に1回読み込む）と生のファイル内容を保持する。
    - このセッションで最初に統合する回答の場合:
       - `## Clarifications` セクションが存在することを確認する（なければ、仕様テンプレートに従い最上位の文脈／概要セクションの直後に作成する）。
       - その配下に（存在しなければ）本日の `### Session YYYY-MM-DD` という小見出しを作成する。
    - 受け入れ直後に箇条書きの行を追記する: `- Q: <question> → A: <final answer>`。
    - 次に、ただちに明確化を最も適切なセクションに適用する:
       - 機能的な曖昧さ → Functional Requirements の箇条書きを更新または追加する。
       - ユーザーインタラクション／アクターの区別 → User Stories または Actors サブセクション（存在する場合）を、明確化された役割、制約、シナリオで更新する。
       - データ形状／エンティティ → Data Model を更新する（フィールド、型、関係を追加）。順序を保持し、追加した制約を簡潔に記す。
       - 非機能的な制約 → Success Criteria > Measurable Outcomes に測定可能な基準を追加／変更する（曖昧な形容詞を指標または明示的な目標に変換する）。
       - エッジケース／異常系フロー → Edge Cases / Error Handling 配下に新しい箇条書きを追加する（テンプレートにそのプレースホルダがあれば、そのサブセクションを作成する）。
       - 用語の衝突 → 仕様全体で用語を正規化する。必要な場合に限り、一度だけ `(formerly referred to as "X")` を追加して元の用語を残す。
    - 明確化により以前の曖昧な記述が無効になる場合は、重複させずにその記述を置き換える。陳腐化した矛盾するテキストを残さない。
    - 文脈喪失のリスクを最小化するため、統合のたびに仕様ファイルを保存する（アトミックな上書き）。
    - 書式を保持する: 無関係なセクションを並べ替えない。見出し階層を維持する。
    - 挿入する各明確化は、最小限かつテスト可能に保つ（説明的なドリフトを避ける）。

7. 検証（**各**書き込み後および最終パスで実施）:
   - Clarifications セッションには、受け入れた回答ごとにちょうど1つの箇条書きが含まれる（重複なし）。
   - 尋ねた（受け入れた）質問の合計 ≤ 5。
   - 更新されたセクションに、新しい回答が解決するはずだった曖昧なプレースホルダが残っていない。
   - 矛盾する以前の記述が残っていない（無効になった代替選択肢が削除されているかスキャンする）。
   - Markdown 構造が有効である。許可される新しい見出しは `## Clarifications`、`### Session YYYY-MM-DD` のみ。
   - 用語の一貫性: 更新されたすべてのセクションで同じ正規用語が使われている。

8. 更新した仕様を `FEATURE_SPEC` に書き戻す。

9. **仕様品質チェックリストの再検証**（存在する場合）:
   - `FEATURE_DIR/checklists/requirements.md` が存在するか確認する。
   - 存在しない場合は、この手順を黙って省略する。
   - 存在する場合:
     1. チェックリストファイルを読み込む。
     2. すべての GitHub タスクリストのチェックボックス行を特定する — コードフェンスの外にある `- [ ]`、`- [x]`、または `- [X]` に一致する行（大文字小文字を区別せず、ネストされた項目の先頭の空白を許容する）。それ以外の内容（見出し、注記、チェックボックスでない箇条書き、メタデータ）はすべて無視する。
     3. 各チェックボックス行について、現在のマーカー状態（チェック済みか未チェックか）と項目テキストを「変更前スナップショット」リストに記録する。
     4. 各チェックボックス項目を、**更新後**の仕様（手順7で保存したばかりのバージョン）に対して再評価する。
     5. 各チェックボックス項目について、チェック済み／未チェックの状態が実際に変わる場合のみ更新する:
        - 項目が現在は合格で未チェックだった場合: `[ ]` を `[x]` に変更する。
        - 項目が現在は不合格でチェック済みだった場合: `[x]`/`[X]` を `[ ]` に変更する。
        - 状態が変わらない場合: マーカーをそのままにする（見た目だけの差分を避けるため既存の大文字小文字を保持する）。
     6. 更新したチェックリストファイルを保存する。**状態が変わったチェックボックス行の `[ ]`/`[x]` マーカー部分のみを切り替える。** それ以外のファイル内容 — 見出し、メタデータ、注記、行の順序、空白 — はノイズの多い差分を避けるため変更しないままにする。
     7. 変更前スナップショットと現在の状態を比較し、完了レポート用に3つのリストを算出する:
        - **新たに合格**: 未チェックからチェック済みに変わった項目。
        - **リグレッション**: チェック済みから未チェックに変わった項目。
        - **依然として未チェック**: 未チェックのままの項目。
     8. 変更前／変更後の合格数を「チェック済み／全チェックボックス項目」として記録する（例: "12/16 → 15/16 items passing"）。

挙動ルール:

- 意味のある曖昧さが見つからない場合（またはすべての潜在的な質問が低影響の場合）、次のように応答する: "正式な明確化に値する重大な曖昧さは検出されませんでした。" そして続行を提案する。
- 仕様ファイルが存在しない場合は、ユーザーにまず `/speckit-specify` の実行を指示する（ここで新しい仕様を作成しないこと）。
- 尋ねる質問の合計が5を超えてはならない（単一の質問に対する明確化の再試行は新しい質問としてカウントしない）。
- 機能的な明確さを阻害する欠落でない限り、推測的な技術スタックの質問は避ける。
- ユーザーの早期終了シグナル（"stop"、"done"、"proceed"）を尊重する。
- フルカバレッジのため質問を行わなかった場合は、コンパクトなカバレッジサマリ（全カテゴリ Clear）を出力し、前進を提案する。
- クォータに達したが影響度の高い未解決カテゴリが残っている場合は、それらを根拠とともに Deferred として明示的にフラグする。

優先順位付けのための文脈: $ARGUMENTS

## 必須の実行後フック

**ユーザーに完了を報告する前に、このセクションを必ず完了させること。**

プロジェクトルートに `.specify/extensions.yml` が存在するか確認する。
- 存在しない、または `hooks.after_clarify` 配下にフックが登録されていない場合は、完了レポートへスキップする。
- 存在する場合、それを読み込み `hooks.after_clarify` キー配下のエントリを探す。
- YAML がパースできない、または無効な場合は、フックチェックを黙って省略し完了レポートへ続行する。
- `enabled` が明示的に `false` のフックを除外する。`enabled` フィールドを持たないフックはデフォルトで有効として扱う。
- 残った各フックについて、フックの `condition` 式を解釈または評価しようとは**しない**こと:
  - フックに `condition` フィールドがない、または null/空の場合、そのフックを実行可能として扱う
  - フックが空でない `condition` を定義している場合、そのフックを省略し、条件評価は HookExecutor の実装に委ねる
- フックのコマンド名からスラッシュコマンドを構成する際は、ドット（`.`）をハイフン（`-`）に置き換える。例: `speckit.git.commit` → `/speckit-git-commit`。
- 実行可能な各フックについて、その `optional` フラグに基づき以下を出力する:
  - **必須フック**（`optional: false`） — **各必須フックについて `EXECUTE_COMMAND:` を必ず出力すること**:

    ```
    **自動フック**: {extension}

    ```
  - **任意フック**（`optional: true`）:

## 完了レポート

完了を報告する（質問ループの終了後または早期終了後）:
- 尋ねて回答した質問の数。
- 更新した仕様へのパス。
- 触れたセクション（名前を列挙）。
