# 機能仕様: MCP コネクタの OAuth 認証（claude.ai 接続）

**フィーチャーブランチ**: `002-mcp-oauth-auth`

**作成日**: 2026-06-20

**ステータス**: Draft

**入力**: ユーザーの説明: "claude.ai のリモート MCP コネクタから「認証付き」で wine-record サーバーに接続できるようにする（Figma/Slack のコネクタと同様に、追加時に OAuth ログイン/同意を要求する）"

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - 認証付きでコネクタを接続する (優先度: P1)

ユーザーは claude.ai（Web / モバイル）でカスタムコネクタとして wine-record の URL を登録する。
コネクタ追加時に OAuth のログイン/同意画面が表示され、ログインして同意すると接続が確立する。
以降、ユーザーは record_wine / preview_record / get_jsa_taxonomy をチャットから利用できる。

**この優先度の理由**: これが本機能の核。現状 claude.ai からはサーバーに接続すらできない（エッジ層 SSO に阻まれる）。
認証付き接続が成立して初めて、ホスト稼働した MCP サーバーが実際に「使える」状態になる。

**独立したテスト**: claude.ai でコネクタ URL を登録 → OAuth 同意 → ツール一覧取得まで（record_wine 実行は US2）を
通すことで、認証ゲートの成立を単独で検証できる。

**受け入れシナリオ**:

1. **前提** コネクタ未登録, **操作** claude.ai に wine-record の URL を登録, **結果** OAuth ログイン/同意画面が表示される
2. **前提** OAuth 同意画面, **操作** 正しい資格情報でログインし同意, **結果** コネクタが「接続済み」になり、3 ツールが一覧に表示される
3. **前提** 接続確立済み, **操作** チャットで get_jsa_taxonomy を呼ぶ, **結果** 認証済みリクエストとして成功し語彙が返る

---

### ユーザーストーリー 2 - 認証済みでワインを記録する (優先度: P1)

認証済みユーザーが、ラベル写真から抽出→preview_record で確認→承認→record_wine で保存、までを
claude.ai 経由で端から端まで実行できる。保存内容は永続化され、未認証では一切実行できない。

**この優先度の理由**: 認証はあくまで手段で、目的は「認証された人だけがワインを記録できる」こと。
ここまで通って初めてホスト稼働の価値（既存機能 US1+US2 がリモートで安全に使える）が実証される。

**独立したテスト**: 接続確立済みの状態で record_wine を実行し、永続化先に 1 件増えることを確認する。

**受け入れシナリオ**:

1. **前提** 認証済み接続, **操作** ラベル写真を渡し preview_record→承認→record_wine, **結果** 記録が保存され wineId が返る
2. **前提** 保存完了, **操作** 永続化先を確認, **結果** 記録が 1 件増えている
3. **前提** 認証トークンが期限切れ, **操作** record_wine を呼ぶ, **結果** 認証エラーで拒否され、再認証を促される

---

### ユーザーストーリー 3 - 未認証アクセスの拒否 (優先度: P2)

認証情報を持たない第三者やツールが直接サーバーのツール実行エンドポイントへアクセスしても、
処理は実行されず、標準的な「認証が必要」という応答（と、どこで認証すればよいかの案内）が返る。

**この優先度の理由**: 認証を導入する以上、「認証なしでは確実に弾かれる」ことが保証されなければ意味がない。
これはセキュリティ境界の最低保証であり、回帰してはならない。

**独立したテスト**: トークン無し・不正トークン・期限切れトークンの 3 ケースで実行が拒否されることを確認する。

**受け入れシナリオ**:

1. **前提** トークン無し, **操作** ツール実行エンドポイントへアクセス, **結果** 「認証が必要」と、認証先の所在を示す案内が返る
2. **前提** 不正または期限切れのトークン, **操作** ツール実行, **結果** 拒否され、処理は一切行われない
3. **前提** 別サービス向けに発行されたトークン, **操作** ツール実行, **結果** 受理対象でないとして拒否される（秘匿情報は応答に含めない）

---

### エッジケース

- 認証提供者（IdP）が一時的に応答しないとき、接続/実行はどうなるか? → ユーザーに失敗が分かる形で伝わり、処理は行われない。
- ユーザーが OAuth 同意画面で「拒否」した場合、コネクタは未接続のままで、サーバー側に副作用は残らない。
- アクセストークンが有効期間中に期限切れになった場合、次の操作で再認証が促される（黙って失敗しない）。
- 認証案内（メタデータ）や認証エラー応答に、トークン・鍵・内部構成などの秘匿情報が一切含まれないこと。
- 認証導入後も、既存の入力検証（名前必須・色の許容値・語彙照合・URL 許可）はこれまでどおり機能する。

## 要件 *(必須)*

### 機能要件

- **FR-001**: システムは、ツール実行エンドポイントへの認証されていないアクセスを拒否し、業界標準の「認証が必要」応答（認証先の所在を示す案内を含む）を返さなければならない (MUST)。
- **FR-002**: システムは、認証提供者（IdP）が発行した有効なアクセストークンを提示したリクエストに限り、ツールの実行を許可しなければならない (MUST)。
- **FR-003**: システムは、提示されたトークンが「自分（このサーバー）宛に発行されたもの」であることを検証し、別の相手向けに発行されたトークンを拒否しなければならない (MUST)。
- **FR-004**: システムは、無効・期限切れ・対象不一致のトークンを拒否し、その際に秘匿情報（トークン値・鍵・内部構成）を応答やログに漏らしてはならない (MUST)。
- **FR-005**: システムは、認証情報を機械可読な形で公開し、接続クライアント（claude.ai）が手動設定なしに「どこで・どう認証すればよいか」を発見できるようにしなければならない (MUST)。
- **FR-006**: 接続クライアント（claude.ai）が、事前登録された OAuth クライアント資格情報で接続できなければならない (MUST)。自動クライアント登録（DCR）に依存してはならない（無料の認証提供者では DCR が使えないため、事前登録した資格情報を claude.ai 側に設定して用いる）。
- **FR-007**: ユーザーは、claude.ai のコネクタ追加フローの中で、ログインと同意を一度行うことで接続を確立できなければならない (MUST)。
- **FR-008**: 認証の成立後、ユーザーは既存の 3 ツール（record_wine / preview_record / get_jsa_taxonomy）を従来どおり利用でき、既存の入力検証・永続化の挙動は変わってはならない (MUST)。
- **FR-009**: トークン発行（認証提供者）の役割は外部の専用サービスに委譲し、本システム内に独自の認証サーバーを実装してはならない (MUST)。
- **FR-010**: 認証はアプリケーション層で行い、ホスティング基盤側のアクセス制限（claude.ai が通過できないもの）に依存してはならない。基盤側の制限は、アプリ層認証の導入に伴い解除すること (MUST)。
- **FR-011**: 認証提供者の資格情報・シークレットは環境変数等の安全な場所で管理し、リポジトリ・ログ・エラー応答に含めてはならない (MUST)（憲章「セキュリティと機密情報」）。
- **FR-012**: ロジックを持つ単位（トークン検証・認証メタデータ生成・認証ミドルウェア）は、憲章の TDD 原則に従い先行テストを持たなければならない (MUST)。

### 主要エンティティ *(機能がデータを扱う場合に記載)*

- **アクセストークン**: 認証済みユーザーを表す短命の資格情報。本サーバー宛（対象）であること・有効期限・発行者を検証する対象。値は秘匿。
- **認証メタデータ**: 「このサーバーを使うにはどの認証提供者で認証すべきか」を機械可読に示す公開情報。クライアントの発見に使う。
- **認証提供者（IdP）設定**: 発行者 URL・対象識別子（audience）・必要なスコープ等。シークレットは秘匿、参照情報のみ構成に保持。

## 成功基準 *(必須)*

### 測定可能な成果

- **SC-001**: 認証情報を持たないツール実行リクエストは 100% 拒否され、処理が実行されることは一度もない。
- **SC-002**: OAuth クライアントを一度だけ事前登録すれば、以降ユーザーは claude.ai でコネクタを接続する際、ログイン・同意のみで（トークン手貼りなしに）接続を確立できる。
- **SC-003**: 認証済みユーザーは、ラベル写真からの記録（preview→承認→record_wine）を端から端まで完了でき、永続化が確認できる。
- **SC-004**: 無効・期限切れ・対象不一致のトークンの 3 ケースすべてで実行が拒否され、応答・ログに秘匿情報が含まれない。
- **SC-005**: 既存の自動検証（型・リント・整形・テスト・ビルドの各ゲート）が、本機能導入後もすべて緑のまま維持される。

## 前提

- 認証提供者として、認証サーバーメタデータ公開・PKCE(S256) を備えた外部 IdP（Auth0）を利用する。**無料プランでは DCR が使えない**ため、OAuth クライアントは手動で事前登録し、claude.ai の Advanced settings に client_id/secret を設定する（DCR には依存しない）。
- まずは単一ユーザー/単一テナントで「認証ゲートを通す」ことを目的とし、複数ユーザー間のデータ分離（マルチテナント）は本機能のスコープ外。
- 既存のサーバー（Express + Streamable HTTP・ステートレス）と永続化（ベクトルストア）はそのまま再利用し、認証は HTTP/トランスポート層に追加する。
- claude.ai のホスト版コネクタを主対象とし、所定のコールバック URL を用いる。Claude Code（ローカルループバック）対応は付随的に扱う。
- ホスティング基盤（Vercel）のエッジ層アクセス制限は解除し、保護の責務はアプリ層 OAuth + IdP に移す。基盤設定（iac/）もこの方針に合わせて更新する。
- 画像保存(US3)・確認ウィジェット(widget)・マルチテナントのデータ分離は本機能のスコープ外（別機能）。
