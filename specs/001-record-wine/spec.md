# Feature Specification: ワインの記録（ラベル写真からの登録）

**Feature Branch**: `001-record-wine`

**Created**: 2026-06-17

**Status**: Draft

**Input**: User description: "ラベル写真から飲んだワインを記録する。文字情報を読み取り、ユーザーが正確性を確認し、外観・香り・味わいは定義済み語彙からタップ選択し、記録を永続化して後で見返せる。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ラベル写真から記録の下書きを作って保存する (Priority: P1)

ユーザーはモバイルアプリでワインのラベル写真を提示する。システムはラベルから
文字情報（ワイン名・生産者・産地・ヴィンテージ・輸入業者）を読み取り、確認画面に
事前入力して提示する。ユーザーは誤読を修正し、内容を承認すると記録が永続化される。

**Why this priority**: これ単体で「飲んだワインを記録する」という中心価値が完成する。
表現タグや画像保存が無くても、ワインの基本情報を残すアプリとして成立する MVP。

**Independent Test**: ラベル写真を1枚渡し、抽出結果が確認画面に出ること、誤った
フィールドを直して承認できること、承認後に同じワインを一覧/参照で取り出せることで
検証できる。

**Acceptance Scenarios**:

1. **Given** 読み取り可能なラベル写真, **When** ユーザーが写真を提示する, **Then** 名前・生産者・産地・ヴィンテージ・輸入業者が抽出され確認画面に事前入力される
2. **Given** 抽出結果に誤読がある, **When** ユーザーが該当フィールドを修正して承認する, **Then** 修正後の値で記録が保存される
3. **Given** 承認前の確認画面, **When** ユーザーが承認していない, **Then** 記録は永続化されない（明示承認が保存の前提）
4. **Given** 保存済みの記録, **When** ユーザーがその記録を参照する, **Then** 保存した基本情報が取得できる

---

### User Story 2 - 外観・香り・味わいを定義済み語彙からタップ選択する (Priority: P2)

ユーザーは記録に対して、外観・香り・味わいの表現を、日本ソムリエ協会の定義済み
語彙からタップで選択して付与する。自由入力は行わない。

**Why this priority**: 記録の質を高める中核機能だが、基本情報の記録（US1）が
できていれば段階的に追加できる。

**Independent Test**: 既存の記録に対し、外観・香り・味わいの各カテゴリで語彙一覧が
提示され、タップ選択した表現が記録に保存・反映されることで検証できる。

**Acceptance Scenarios**:

1. **Given** 確認/編集画面, **When** ユーザーが外観・香り・味わいのカテゴリを開く, **Then** 各カテゴリの定義済み表現一覧が提示される
2. **Given** 表現一覧, **When** ユーザーが複数の表現をタップ選択する, **Then** 選択した表現が記録に付与される
3. **Given** 表現選択UI, **When** ユーザーが操作する, **Then** 自由テキスト入力の手段は提供されない

---

### User Story 3 - ラベル画像を永続保存して見返す (Priority: P3)

ユーザーが提示したラベル画像を永続保存し、後で記録の詳細からその画像を参照できる。

**Why this priority**: 見返しの体験を高めるが、記録そのもの（US1/US2）とは独立して
追加可能。

**Independent Test**: 記録時にラベル画像を保存し、後で同じ記録の詳細から画像を
取得・表示できることで検証できる。

**Acceptance Scenarios**:

1. **Given** ラベル画像付きの記録操作, **When** ユーザーが記録を承認する, **Then** ラベル画像が永続保存され記録に紐づく
2. **Given** 画像付きの保存済み記録, **When** ユーザーが詳細を参照する, **Then** 保存されたラベル画像を取得できる

---

### Edge Cases

- ラベルが部分的にしか読めない / 文字が判読不能なとき、抽出できたフィールドだけ
  事前入力し、残りは空欄でユーザー入力（タップ/選択）に委ねる。
- ヴィンテージが無い（ノン・ヴィンテージ）ワインのとき、年を「NV」として扱える。
- 同一ワインを再度記録しようとしたとき（重複の可能性）も、保存自体はブロックしない
  （重複検知は本スコープ外、将来の検索機能で扱う）。
- ユーザーが承認せずに離脱したとき、下書きは永続化されない。
- 画像の永続保存に失敗したとき（US3）、基本情報の記録（US1）は成立させ、画像欠落を
  ユーザーに通知する。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムはラベル画像から文字情報（ワイン名・生産者・産地・ヴィンテージ・輸入業者）を抽出しなければならない。
- **FR-002**: システムは抽出結果を、保存前にユーザーが確認・修正できる形で提示しなければならない。
- **FR-003**: システムはユーザーの明示的な承認があって初めて記録を永続化しなければならない。
- **FR-004**: システムは外観・香り・味わいの表現を、定義済み語彙からの選択肢として提示しなければならない（カテゴリ別）。
- **FR-005**: ユーザーは表現を複数タップ選択でき、システムは自由テキスト入力の手段を表現入力に提供してはならない。
- **FR-006**: システムは記録を永続化し、後で参照・取得できなければならない。
- **FR-007**: システムは記録に紐づくラベル画像を永続保存し、後で取得できなければならない。
- **FR-008**: システムはヴィンテージ無し（NV）を表現・保存できなければならない。
- **FR-009**: システムは産地を階層（国・地方・地区・村など判明する範囲）として保持できなければならない（後続の観点別検索の基盤）。

### Key Entities *(include if feature involves data)*

- **ワイン記録 (WineRecord)**: 1本のワインの記録。名前・生産者・産地（階層）・
  ヴィンテージ・輸入業者・購入店舗・外観/香り/味わいの表現タグ・ラベル画像参照・
  記録日時を持つ。
- **表現語彙 (Expression Taxonomy)**: 日本ソムリエ協会の定義済み表現の集合。
  外観・香り・味わいのカテゴリごとにタームを持つ、選択肢の供給源。
- **ラベル画像 (LabelImage)**: 永続保存されたラベル画像と、その参照（URL）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーは1本のワインを、写真提示から保存完了まで2分以内に記録できる。
- **SC-002**: 表現の付与は最後まで文字入力ゼロ（すべてタップ選択）で完了できる。
- **SC-003**: 読み取り可能なラベルで、抽出される基本フィールドの過半（5項目中3項目以上）が、ユーザー修正なしに正しい。
- **SC-004**: 保存した記録は、基本情報・表現タグ・ラベル画像のすべてを後で欠落なく取得できる。
- **SC-005**: ユーザーが承認しなかった下書きは100%永続化されない。

## Testing

本機能は憲章の原則 II（Test-First, NON-NEGOTIABLE）に従い、テストを先に書く。
タスク生成時にユーザーストーリーごとのテストタスクを含める（受け入れシナリオを
検証可能な形に落とす）。

## Assumptions

- 文字情報の読み取り（OCR）は、接続先のマルチモーダル LLM（Claude / Codex）の
  画像読解能力で行う前提（専用 OCR サービスは置かない）。
- まずは個人利用のスケール（単一ユーザー、数千件規模）を想定する。
- 定義済み表現語彙は、別途提供される日本ソムリエ協会の表現集（PDF）を構造化した
  データとして与えられる前提（その構造化は前提タスクとして扱う）。
- 検索・観点別提示は本スコープ外（次スコープ）。本スコープは記録フローに限定する。
- まず Claude モバイルでの動作を主対象とし、Codex は後追いで検証する。
