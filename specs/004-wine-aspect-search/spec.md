# 機能仕様: 観点独立のワイン類似検索

**フィーチャーブランチ**: `004-wine-aspect-search`

**作成日**: 2026-06-24

**ステータス**: ドラフト

**入力**: ユーザー記述: "記録済みのワインから、外観・香り・味わいを自然文で表現した条件に意味的に近いワインを探す。観点（外観/香り/味わい）ごとに独立して近さを測り、産地・ヴィンテージ・色は構造的な絞り込みとして扱う。例: 『外観は澄んだ輝きのある、味わいはビロードのようなタンニンを感じるスペインの赤ワイン』"

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー1 - 単一観点の表現で似たワインを探す（優先度: P1）

ユーザーは「味わいがビロードのようなタンニンのワイン」のように、ひとつの観点（外観・香り・味わいのいずれか）について表現を述べる。システムはその観点の表現の**意味的な近さ**だけでワインを順位づけし、上位のワインを表示情報つきで返す。

**この優先度の理由**: これ単体で「ある観点の表現が似たワインを引く」という中心価値が完成する。複数観点の合成や構造的絞り込みが無くても、観点別の意味検索として成立する MVP。憲章 原則 V（縦切り）に沿い、まず単一観点で端から端まで動かす。

**独立したテスト**: いくつかの記録があるコーパスに対し、ひとつの観点表現を渡し、その観点が近いワインが上位に来ること、各結果に表示情報とスコアが付くことで検証できる。

**受け入れシナリオ**:

1. **前提** 味わいの表現が記録された複数のワイン, **操作** ユーザーが味わいの表現（例: なめらかなタンニン）で検索する, **結果** 味わいが意味的に近いワインが上位に順位づけされて返る
2. **前提** 検索結果, **操作** ユーザーが結果を見る, **結果** 各件に表示情報（名前・生産者・産地・ヴィンテージ・色・画像参照）と近さスコアが含まれる
3. **前提** 検索した観点（味わい）の表現を持たないワイン, **操作** 同じ味わい検索を行う, **結果** その観点を持たないワインはその観点の一致として扱われない（上位に来ない）

---

### ユーザーストーリー2 - 複数観点を組み合わせ、観点ごとに独立して評価する（優先度: P2）

ユーザーは「外観は澄んだ輝きのある、味わいはビロードのようなタンニン」のように複数観点を同時に述べる。システムは**各観点をそれぞれの観点の中だけで**近さを測り（観点をまたいで表現を混ぜない）、観点別スコアを合成して総合順位を決める。任意で観点ごとの重み（例: 味わい重視）を反映できる。

**この優先度の理由**: 「観点独立検索」の差別化価値が完全に立ち上がる中核。単一観点（US1）の上に、観点別スコアの合成という独立した能力を積む。

**独立したテスト**: 外観と味わいの両方を指定したクエリで、両観点を満たすワインが、片方しか満たさないワインより上位に来ること、結果に観点別の内訳スコアが出ることで検証できる。

**受け入れシナリオ**:

1. **前提** 外観・味わいが記録された複数のワイン, **操作** ユーザーが外観句と味わい句を同時に指定して検索する, **結果** 両観点で近いワインが、片方のみ近いワインより上位に来る
2. **前提** 複数観点クエリ, **操作** ユーザーが味わいを重視する重みを指定する, **結果** 味わいの近さが総合順位により強く反映される
3. **前提** 複数観点クエリの結果, **操作** ユーザーが各件を見る, **結果** 総合スコアに加え観点別（外観・味わい）の内訳スコアが提示される
4. **前提** 外観句と味わい句, **操作** システムが評価する, **結果** 外観句は外観の表現としてのみ、味わい句は味わいの表現としてのみ評価され、観点をまたいで混ざらない

---

### ユーザーストーリー3 - 産地・ヴィンテージ・色で構造的に絞り込む（優先度: P3）

ユーザーは観点表現に加えて「スペインの赤ワイン」「2019年の」のような産地・色・ヴィンテージの条件を述べる。システムはこれらを**意味的な近さではなく構造的な絞り込み**として適用し、条件に合致するワインの中だけで観点別の意味検索を行う。

**この優先度の理由**: 検索の実用性を高めるが、観点別の意味検索（US1/US2）が動いていれば独立して追加できる。憲章 原則 IV（構造的近接をベクトル距離で代替しない）の具体化。

**独立したテスト**: 産地=スペイン・色=赤の条件を付けたクエリで、結果がすべてその条件を満たすこと、条件を外すと候補が広がることで検証できる。

**受け入れシナリオ**:

1. **前提** 産地の異なる複数のワイン, **操作** ユーザーが産地=スペインを条件に加えて検索する, **結果** 返るワインはすべて産地がスペインに合致する
2. **前提** 構造条件に合致するワインが存在しない, **操作** ユーザーがその条件で検索する, **結果** エラーではなく空の結果が返る
3. **前提** 産地・色の条件のみ（観点表現なし）, **操作** ユーザーが検索する, **結果** 条件に合致するワインの一覧が返る（意味順位は付かない）

---

### エッジケース

- 観点表現も構造条件も**何も指定されない**とき、検索は成立しないためフィールド別のエラーを返す（最低ひとつの観点表現または構造条件が必要）。
- 統制語彙（JSA）の**完全一致でない言い換え表現**（例: 「ビロードのような」「土っぽい」）でも、意味的に近い表現を持つワインを拾えなければならない。
- 指定した観点の表現を持たないワインは、その観点の一致として数えない（合成時にその観点の寄与を持たない）。
- 返却件数の上限が記録総数を超えるとき、合致した全件を返す。
- スコアが同点のとき、順序は決定的（再実行で同じ順序）でなければならない。
- 検索対象のワインが0件（コーパスが空）のとき、エラーではなく空の結果を返す。
- 検索の起点として渡された下書き／クエリは、記録を一切作成・変更しない（読み取り専用）。

## 要件 *(必須)*

### 機能要件

- **FR-001**: システムは、指定された観点（外観・香り・味わい）ごとに、その観点の表現の**意味的な近さを独立して**評価しなければならない（観点をまたいで表現を混合してはならない）。〔原則 IV〕
- **FR-002**: 複数の観点が指定された場合、システムは各観点の近さスコアを合成して総合順位を決定しなければならない。
- **FR-003**: システムは観点ごとの重みづけを任意で受け付け、合成に反映できなければならない（未指定時は観点間で均等）。
- **FR-004**: システムは産地（国・地方・地区・村）・ヴィンテージ・色を、意味的な近さではなく**構造的な絞り込み条件**として適用しなければならない。構造的近接を意味的距離で代替してはならない。〔原則 IV〕
- **FR-005**: システムは、統制語彙（JSA）の完全一致ではない言い換え表現でも、意味的に近い表現を持つワインを検索できなければならない。
- **FR-006**: システムは各結果について、総合スコアに加え**観点別の内訳スコア**を返さなければならない（順位の根拠の透明性）。
- **FR-007**: システムは各結果について、表示に必要な記録情報（名前・生産者・産地・ヴィンテージ・色・画像参照など）を含めて返さなければならない。
- **FR-008**: システムは返却件数の上限を入力として受け付け、未指定時は妥当な既定値を用いなければならない。
- **FR-009**: システムは入力（観点表現・構造条件・件数・重み）を境界で検証し、不正な場合はどのフィールドがなぜ不正かを構造化して返さなければならない。〔原則 I / III〕
- **FR-010**: 検索は読み取り専用で、記録の作成・更新・削除などの副作用を持ってはならない。
- **FR-011**: 同一クエリ・同一コーパスに対する検索結果は決定的（同じ順位）でなければならない（同点時の順序も安定）。
- **FR-012**: 観点表現が一切なく構造条件のみが指定された場合、システムは構造条件に合致するワインを返さなければならない（意味的な順位づけは行わない）。

### 主要エンティティ *(データを扱う場合)*

- **検索クエリ (SearchQuery)**: ひとつの検索要求。観点別の表現テキスト（外観・香り・味わい、いずれも任意）、構造的な絞り込み条件（産地階層・ヴィンテージ・色、いずれも任意）、返却件数、観点ごとの重み（任意）を持つ。
- **検索結果項目 (SearchResultItem)**: 1件のヒット。対象ワインの識別子、表示用の記録情報、総合スコア、観点別の内訳スコアを持つ。
- **ワイン記録 (WineRecord)**: 既存エンティティ（001）を参照。検索は記録済みの表現・産地階層・色・ヴィンテージ・表示情報を読み取り対象とする。

## 成功基準 *(必須)*

### 測定可能な成果

- **SC-001**: 単一観点の表現クエリで、その観点が近いワインが上位5件に並び、ユーザー評価で過半（3/5以上）を妥当と判断できる。
- **SC-002**: 複数観点クエリで、両観点を満たすワインが、片方のみを満たすワインより必ず上位に来る。
- **SC-003**: 構造条件（例: 産地・色）を付けた検索結果は、100% その条件に合致する（条件外の混入ゼロ）。
- **SC-004**: 統制語彙の完全一致語を使わない言い換え表現での検索でも、関連するワインが結果に含まれる（用意した言い換えクエリ群の8割以上で関連ワインを返す）。
- **SC-005**: 数千件規模のコーパスで、ユーザーは検索結果を体感的に即時（1秒以内）に受け取れる。
- **SC-006**: 同一クエリを再実行したとき、結果の順位は100%同一である（決定性）。

## テスト

本機能は憲章 原則 II（テスト駆動、交渉の余地なし）に従い、テストを先に書く。とりわけ観点別ランキングの独立性（FR-001）・複数観点の合成順位（FR-002/003）・構造的絞り込みの厳密性（FR-004）・決定性（FR-011）は、受け入れシナリオを検証可能なテストに落としてから実装する。タスク生成時にユーザーストーリーごとのテストタスクを含める。

## 前提

- 検索対象は単一ユーザーの記録（数千件規模）。マルチユーザーや大規模スケールは本スコープ外。〔001 の前提と一貫〕
- ユーザーの自然文（「外観は…味わいは…スペインの赤」）を、観点別の表現と構造条件に**分解する工程は接続先の LLM が担い**、検索ツールは分解済みの構造化入力を受け取る前提。ツール自体は自然文パーサを持たない。〔原則 I：契約の明確さ〕
- 観点別スコアの合成は、観点間の重み付き平均を既定とする（重み未指定時は均等）。具体的な合成式・正規化はプラン段階で確定する妥当なデフォルトとして扱う。
- 「特定の記録に似たワインを探す（記録 ID を起点にした類似検索）」は本スコープ外（自然文表現を起点とする検索に限定）。自然な将来拡張として記録しておく。
- 検索結果の表示情報は既存の記録（正本）から復元する。記録のスキーマ（001）に依存する。
- 検索フローの提示 UI（タップ絞り込み等）は、リモート HTTP のウィジェット制約（001 R5）が解消するまではテキストベースで成立させる。
