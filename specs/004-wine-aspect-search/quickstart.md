# クイックスタート / 検証ガイド: 観点独立のワイン類似検索

`search_wines` がエンドツーエンドで動くことを示す手順。実装詳細は tasks.md と実装フェーズに属する。
契約は [contracts/search-wines-tool.md](./contracts/search-wines-tool.md)、設計は [plan.md](./plan.md)/[research.md](./research.md)。

## 前提

- 001 の記録フローが稼働し、Upstash Vector に `overall` ＋観点別 namespace が索引済み。
- 検証用に、観点表現（外観/香り/味わい）と産地・色・ヴィンテージが異なる複数のワインを記録しておく
  （例: スペイン赤・フランス白・日本赤 など、味わい/外観の表現が分かれるもの）。
- ローカル env（`UPSTASH_VECTOR_REST_URL`/`_TOKEN`）。埋め込みインデックスは bge-m3。

## 自動ゲート（CI/ローカル・env 非依存・原則 II）

実装の先行テストで以下が緑になること（フェイク `VectorStore` で検証）:

- **C2/SC-002**（合成順位）: 外観+味わい均等重みで、両観点 0.8 のワイン（score 0.8）が味わいのみ 0.9 のワイン
  （score 0.45）より上位。← `combineScores.test.ts`（**先に失敗させてから実装**）
- **C1/FR-001**（観点独立）: 外観句は appearance namespace のみ、味わい句は taste namespace のみに渡る。
- **C4/SC-003**（構造 exact）: color=red・country=スペイン 指定で、条件外のワインが結果に 0 件。
- **C11/SC-006**（決定性）: 同点スコアを含むコーパスで、再実行しても順位が 100% 同一（wineId tiebreak）。
- **C12/FR-012**（構造のみ）: 観点表現なし・color=red で、赤のみが wineId 昇順・スコアなしで返る。
- **C9/FR-009**（検証）: 観点も構造も空 → フィールド別エラー。weights 負値・limit<1 → エラー。
- 既存テスト＋全ゲート（typecheck/lint/format/test/build）緑。

```sh
npm run typecheck && npm run lint && npm run format:check && npm run test && npm run build
```

## 実機検証（手動・本機能の受け入れ）

claude.ai（または dev サーバ）から `search_wines` を呼ぶ。自然文は LLM が構造化入力へ分解する。

1. **US1 / SC-001（単一観点）**: 「味わいがなめらかなタンニンのワイン」→ `{taste:"なめらかなタンニン"}`。
   味わいが近いワインが上位5件に並び、各件に表示情報＋スコアが付く。過半を妥当と判断できる。
2. **US2 / SC-002（複数観点）**: 「外観は澄んだ輝きのある、味わいはビロードのようなタンニン」→
   `{appearance:"澄んだ輝きのある", taste:"ビロードのようなタンニン"}`。両観点を満たすワインが、片方のみの
   ワインより上位。各件に観点別の内訳スコアが出る。重み `{taste:2}` で味わい寄りに順位が動く。
3. **US3 / SC-003（構造絞り込み）**: 上記に `{color:"red", region:{country:"スペイン"}}` を追加 → 結果は全件
   スペイン赤。条件を外すと候補が広がる。条件のみ（観点表現なし）でも一覧が返る（順位なし・FR-012）。
4. **SC-004（言い換え）**: JSA 完全一致でない表現（「土っぽい」「ビロードのような」）で関連ワインが返る
   （用意した言い換えクエリ群の 8 割以上）。
5. **SC-005（即時性）**: 数千件規模でも体感 1 秒以内。
6. **エッジ**: 観点も構造も空 → エラー。空コーパス／合致ゼロ → 空結果（エラーにしない）。

## 完了の定義

- 自動ゲート（C1/C2/C4/C9/C11/C12）が全部緑。
- 実機で US1〜US3 が成立し、観点独立・構造 exact・決定性・内訳スコアを確認。
- 検索が読み取り専用（記録の作成・変更が起きない・FR-010）。
