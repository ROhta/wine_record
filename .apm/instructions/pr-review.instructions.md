# Pull Request レビューについて

## 確認観点の優先順位

PR をレビューする際は、**PR 本文の `## 見てほしいところ` を最優先で確認する**。

- まず PR 本文（`gh pr view <pr> --json body -q .body` 等）を取得し、`## 見てほしいところ` の各項目を抽出する。コードを読む前に着眼点としてチェックリスト化する。
- 当該項目への所見（妥当 / 懸念あり / 追加質問）を、他のレビュー指摘より先にコメントとして残す。
  - **例外**: セキュリティ脆弱性・データ損失・致命的バグ・本番障害につながる問題など緊急度の高い指摘は、通常どおり先に提示してよい。
- `## 見てほしいところ` が次のいずれかなら「実質空」とみなし、その旨を作者に確認してから一般観点のレビューに進む。
  - 節自体が存在しない
  - チェックボックス項目のみで、それ以降の本文が空白文字しかない
  - PR テンプレートの初期値のまま編集されていない

理由: ここは作者が特に意見・確認を求めている箇所であり、後回しにすると「作者の関心事に答えていない」レビューになりやすい。先にここを潰してから自由観点の指摘に移ることで、レビューの価値が安定する。

## コミュニケーション

- レビューコメント・返信は、いかなる場合でも**日本語**で記述する。
- 指摘は具体的に書く（該当箇所・理由・可能なら修正案）。
